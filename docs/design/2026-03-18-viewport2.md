# Design Document: viewport2

**Author:** Scott A. Idler
**Date:** 2026-03-18
**Status:** Implemented
**Review Passes Completed:** 5/5

## Summary

viewport2 is a Linux desktop tool that displays a draggable, resizable overlay frame on screen and outputs the captured region inside that frame as a virtual camera feed via v4l2loopback. This lets a 4K desktop user share a laptop-sized portion of their screen in Google Meet or Slack without overwhelming the audience.

## Problem Statement

### Background

Presenting from a 4K desktop in video calls (Meet, Slack) forces a choice between sharing the entire 4K display (too much content, tiny text for viewers) or sharing a single window (too restrictive, can't show multiple things or switch contexts fluidly). There is no good middle ground on Linux.

### Problem

There is no tool on Linux that lets you define an arbitrary, movable region of your screen and share just that region as a camera feed in any video conferencing app.

### Goals

- Provide a draggable, resizable overlay frame visible on the user's desktop
- Capture the screen content inside the frame in real-time
- Output the captured frames to a v4l2loopback virtual camera device
- Work with Google Meet and Slack (which support camera selection)
- Support the user's environment: Ubuntu 25.10, GNOME/Mutter, Xwayland

### Non-Goals

- Audio capture - video only
- Windows/macOS support
- Recording to file (use OBS for that)
- Streaming to anything other than v4l2loopback
- Supporting pure X11 (non-Wayland) setups

## Proposed Solution

### Overview

A single Rust binary with three subsystems:

1. **Overlay** - a GTK4 window that renders as a hollow frame (transparent interior, colored border) that is always-on-top, draggable, and resizable
2. **Capture** - uses XDG Desktop Portal + PipeWire to capture the full screen, then crops to the overlay's current position/size
3. **Output** - writes cropped frames to a v4l2loopback device as a virtual camera

### Shared State

The overlay and the capture pipeline run on different threads. They communicate via a shared `AtomicRect` (x, y, width, height as AtomicU32 values) that the overlay updates on drag/resize and the pipeline reads each frame to determine the crop region.

### Module Breakdown

**`cli.rs`** - CLI argument parsing
- `--device <path>` - v4l2loopback device (default: `/dev/video10`)
- `--size <WxH>` - initial frame size (default: `1280x720`)
- `--fps <N>` - target frame rate (default: `30`)
- `--color <hex>` - border color (default: `#ff3333`)
- `--border-width <px>` - border thickness (default: `4`)

**`overlay.rs`** - GTK4 overlay window
- Transparent window with CSS-based hollow frame (border only, transparent fill)
- Always-on-top via GTK4 window hints
- Draggable by clicking anywhere on the border
- Resizable via edge/corner drag handles
- Displays current dimensions in a subtle label
- Keyboard shortcuts:
  - `Escape` - quit
  - `Arrow keys` - nudge position by 10px (+ Shift for 1px)
  - `Ctrl+Arrow` - resize by 10px
  - `1` through `4` - preset sizes (1280x720, 1920x1080, 960x540, custom from config)

**`capture.rs`** - screen capture via PipeWire
- Uses `ashpd` to negotiate a screencast session with the XDG portal
- Requests `PersistMode::ExplicitRevoke` so the user only approves once
- Stores the restore token in the config file for reuse across sessions
- Connects to PipeWire, attaches to the stream
- Receives full-screen frames as SPA buffers (SHM preferred, DMA-BUF as fallback)

**`output.rs`** - v4l2loopback writer
- Opens the device for writing
- Sets format via `VIDIOC_S_FMT` (YUYV, fixed output resolution from config)
- Each frame: crop the full-screen buffer to the overlay rect, scale to output resolution, convert BGRx to YUYV, write

**`convert.rs`** - pixel format conversion
- BGRx (from PipeWire) to YUYV (for v4l2loopback)
- Uses `yuvutils-rs` for SIMD-accelerated conversion

**`pipeline.rs`** - ties capture + convert + output together
- Runs on a dedicated thread
- Reads the shared rect each frame
- Output resolution is fixed (from config, default 1280x720) - overlay resize scales to fit, avoiding v4l2 format renegotiation

### Data Flow Per Frame

1. PipeWire delivers a full-screen frame (BGRx, e.g., 3840x2160)
2. Read current overlay rect from shared state (e.g., x=500, y=200, w=1280, h=720)
3. Crop the frame buffer to that rect (zero-copy row pointer math)
4. Convert cropped BGRx to YUYV via yuvutils-rs
5. Write YUYV buffer to `/dev/videoN`
6. Target: complete in <33ms for 30fps

### Config File (`~/.config/viewport2/viewport2.yml`)

Config lives in the standard XDG config directory. CLI args override config values.

```yaml
device: /dev/video10
output_size: 1280x720    # fixed output resolution to virtual camera
initial_size: 1280x720   # initial overlay size on screen
fps: 30
border_color: "#ff3333"
border_width: 4
portal_restore_token: ~   # auto-populated after first portal approval
presets:
  - 1280x720
  - 1920x1080
  - 960x540
```

## Alternatives Considered

### Alternative 1: X11 XShmGetImage capture
- **Description:** Bypass PipeWire entirely, use X11 shared memory extension to capture screen regions directly
- **Pros:** Simpler, no portal dialog, lower latency, no PipeWire dependency
- **Cons:** Only works under Xwayland, may not capture Wayland-native windows, not future-proof as GNOME moves toward pure Wayland
- **Why not chosen:** PipeWire portal is the sanctioned approach on Wayland and handles all window types. The portal persistence token eliminates the approval friction after first use.

### Alternative 2: GStreamer pipeline
- **Description:** Use GStreamer's `pipewiresrc` + `videocrop` + `v4l2sink` for the media pipeline
- **Pros:** Battle-tested media handling, automatic format negotiation, less custom code
- **Cons:** Heavy dependency, less control over latency, harder to integrate tightly with the overlay's dynamic rect
- **Why not chosen:** For a single-purpose tool, the direct PipeWire + v4l2 path is lighter and gives us full control over the frame-by-frame pipeline.

### Alternative 3: OBS Studio virtual camera
- **Description:** Use OBS with a screen capture source + crop filter + virtual camera output
- **Pros:** Already exists, no development needed
- **Cons:** No draggable overlay - repositioning requires editing the crop filter in OBS UI. Heavyweight. Not a smooth UX for the use case.
- **Why not chosen:** The UX of dragging a frame is the core value proposition.

### Alternative 4: Zoom "Share Portion of Screen"
- **Description:** Zoom has this feature built in
- **Pros:** Polished UX, exactly the right interaction model
- **Cons:** Only works in Zoom. User needs Meet and Slack support.
- **Why not chosen:** Vendor-locked to Zoom.

## Technical Considerations

### Dependencies

**System packages (build-time):**
- `libgtk-4-dev` - GTK4 development headers
- `libpipewire-0.3-dev` - PipeWire development headers

**System packages (runtime):**
- `v4l2loopback-dkms` - kernel module (already installed)
- PipeWire (already running)
- XDG Desktop Portal (comes with GNOME)

**Rust crates:**
- `gtk4` - overlay UI
- `ashpd` - XDG portal screencast negotiation
- `pipewire` - PipeWire stream client
- `yuvutils-rs` - SIMD pixel format conversion
- `nix` - ioctl for v4l2 device setup
- `clap` - CLI parsing (from scaffold)
- `eyre` - error handling (from scaffold)
- `serde`/`serde_yaml` - config (from scaffold)

### Performance

- Target: 30fps at 1280x720 output
- BGRx to YUYV conversion for 1280x720: ~0.5ms with SIMD (yuvutils-rs)
- Crop is zero-copy (pointer arithmetic on the full-screen buffer)
- PipeWire frame delivery latency: typically <5ms
- Total per-frame budget: 33ms, expected usage: <10ms
- CPU overhead: minimal - one thread doing crop+convert+write, GTK main thread idle most of the time

### Security

- Portal handles screen capture permission (user must approve)
- v4l2loopback device permissions: user must be in `video` group or device must have appropriate udev rules
- No network access, no elevated privileges needed after initial `modprobe`

### Testing Strategy

- **Unit tests:** pixel conversion correctness, rect math, config parsing
- **Integration tests:** v4l2loopback write (requires loaded module), portal session lifecycle (requires running compositor - skip in CI)
- **Manual testing:** visual verification of overlay, frame content, Meet/Slack camera selection

### Rollout Plan

1. Install dev dependencies (`libgtk-4-dev`, `libpipewire-0.3-dev`)
2. Implement in phases (see Implementation Plan)
3. Install binary via `cargo install --path .`
4. Setup: `sudo modprobe v4l2loopback devices=1 video_nr=10 card_label="Viewport" exclusive_caps=1`
5. Run: `viewport2 --device /dev/video10`

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| GNOME/Mutter doesn't support layer-shell for always-on-top overlay | N/A | N/A | Not using layer-shell. Using GTK4 `set_keep_above()` which works on GNOME via Xwayland hints. |
| PipeWire delivers DMA-BUF frames that are hard to read on CPU | Medium | Medium | Negotiate SHM format preference in PipeWire stream params. Fall back to GPU map via `gbm` if needed. |
| Portal restore token doesn't persist across reboots | Low | Low | Gracefully handle re-prompting. Store token in config file. |
| v4l2loopback format rejected by Meet/Slack | Low | High | Use YUYV which is the most universally supported format. Test with both apps early. |
| Overlay captures itself in the PipeWire stream | Medium | Medium | The overlay border is outside the capture rect. Interior is transparent so the compositor renders what's underneath. PipeWire captures the composited output which includes what's under the transparent area. |
| High CPU usage on 4K source with 30fps crop+convert | Low | Medium | Crop is zero-copy. YUYV conversion is SIMD. Profile early. If needed, reduce capture resolution via PipeWire negotiation. |
